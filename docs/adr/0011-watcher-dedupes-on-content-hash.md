# Vault watcher dedupes on content hash, not timing

The Vault watcher distinguishes "our own write" from "external edit" by comparing the file's current sha256 against a per-path cache. The cache is populated two ways: by `markSelfWrite(absPath)` immediately after a writer commits bytes (sync read of the just-written file), and by `onFlush` itself when it lets an event through (re-hash, update). An event whose file content matches the cached hash is a no-op — no re-index, no SSE publish.

The earlier design relied on a time-based suppression window: `markSelfWrite` would block events for the same path for 1 second. That worked on a quiet filesystem and broke immediately on macOS with iCloud Drive: `Documents/` is iCloud-synced, so xattrs (`com.apple.metadata:*`, `com.apple.macl`) are mutated by the OS 2–4 seconds after we write. Each xattr touch fires a `change` event with our just-written bytes; past TTL, it leaked through as a "this Note changed externally" notification, and with a dirty editor buffer the client surfaced the conflict modal mid-keystroke. Extending the TTL was rejected because the right value is "however long iCloud and Spotlight take" — unbounded — and any window large enough to swallow xattr noise also swallows real external edits for the same duration.

Content-hash dedupe is timing-independent. iCloud xattrs, FSEvents echoes, and the second-burst events macOS emits for tmp-and-rename all carry identical bytes; they collapse to no-ops. A genuine external edit (Obsidian, vim, `sed -i`) changes the bytes and flows through to re-index + publish. The cost is one sync sha256 per `markSelfWrite` and one async sha256 per `onFlush` — for typical Note files (~kilobytes) the overhead is irrelevant against the I/O the watcher was already doing. The convention this imposes on writers — *call `markSelfWrite` after the write completes, not before* — is also the simpler ordering; `renameNote` was the one outlier and was refactored to match.
